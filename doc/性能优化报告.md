版本：v1.1 | 日期：2026-05-10 | 硬件：Intel i9-12900K, 12P+4E Core, DDR4-3200 双通道

## 二、根因分析（为什么 Phase 0 只有 2.73 tok/s）

### 2.1 内存带宽是 decode 的真正瓶颈

GPT-2 117M 单 token decode 需读取的权重量（F16）：

| 操作 | 读取量 |
|------|--------|
| 12×QKV projection | 42.5 MB |
| 12×Attention proj | 14.2 MB |
| 12×FFN up-proj   | 56.6 MB |
| 12×FFN down-proj | 56.6 MB |
| LM head (wte.T)  | 77.2 MB |
| **合计**          | **247 MB/token** |

理论带宽上限（DDR4-3200 双通道）：51.2 GB/s → **207 tok/s**  
计算强度 ≈ 1 FLOP/byte（典型内存受限场景）

Phase 0 实际带宽 = 2.73 × 247 = **674 MB/s = 1.3%** → 严重利用不足

### 2.2 三大根因

**根因 1：标量 GEMV，每元素需要位操作**  
Phase 0 的 `f16_to_f32` 对每个权重做 3 条分支 + bit-manipulation，无法被编译器自动向量化：
```cpp
// Phase 0 - 标量，约 0.5 elem/cycle
for (int k = 0; k < in; k++)
    acc += f16_to_f32(row[k]) * x[k];
```

**根因 2：单线程**  
i9-12900K 有 12 P-core，全部闲置。多线程能同时发起多个内存流，提高 DRAM 利用率。

**根因 3：每 token 264 次热路径 `snprintf` + 哈希查找**  
12 层 × (4 个权重名 LayerNorm + 4 个 GEMV) × 2次/次 = 264次。在 ~1 tok/s 级别不明显，但到 50+ tok/s（每 token 13ms）时占比可达 2%。

## 三、优化设计详解

### 3.1 设计一：分离编译 AVX2+F16C GEMV 内核

**文件**：`src/backend/cpu_x86/kernels/gemv_f16_avx2.cpp`  
**编译标志**：`-mavx2 -mfma -mf16c -O3`（独立 CMake OBJECT 库）

核心技术：F16C 指令 `_mm256_cvtph_ps`——单条指令将 8 个 F16 转为 8 个 F32，配合 FMA3 的 `_mm256_fmadd_ps`（0.5 cycle 吞吐）：

```cpp
// 双路展开：两个独立 accumulator 隐藏 cvtph_ps 的 ~6 cycle 延迟
for (; k + 16 <= K; k += 16) {
    const __m256 w0 = _mm256_cvtph_ps(_mm_loadu_si128(...W + k));
    const __m256 w1 = _mm256_cvtph_ps(_mm_loadu_si128(...W + k + 8));
    acc0 = _mm256_fmadd_ps(w0, _mm256_loadu_ps(x + k),     acc0);
    acc1 = _mm256_fmadd_ps(w1, _mm256_loadu_ps(x + k + 8), acc1);
}
```

**关键设计决策：分离编译单元（TU）**

`gpt2_model.cpp` 不编译为 AVX2 目标，只做前向声明 + 函数指针选择。避免「整个模型文件都需要 `-mavx2`」的污染：

```cpp
// 仅前向声明（不需要 SIMD 编译标志）
namespace axonforge::cpu_x86 {
    void gemv_f16_avx2_range(float* y, const uint16_t* W, ...);
}
// 运行时一次性选择
static const GemvRangeFn kGemvRange = cpu_features().avx2
    ? cpu_x86::gemv_f16_avx2_range
    : gemv_f16_scalar_range;
```

吞吐：16 elements / 3 cycles ≈ **5.3 elem/cycle**（vs 标量 0.5 elem/cycle，提升 **10.6×**）

---

### 3.2 设计二：持久 Fork-Join 线程池

**文件**：`src/core/thread_pool.hpp`（header-only，~100 行）

**为什么不用 OpenMP？**

| | OpenMP | 自研持久 ThreadPool |
|--|--------|---------------------|
| 线程生命周期 | 每次 `parallel for` fork/join | 初始化时创建，永久存活 |
| 唤醒开销 | ~5–30 μs/call（OS 线程创建） | ~500 ns/call（条件变量唤醒） |
| GPT-2 GEMV 次数/token | 264 | 264 |
| 累计开销 @50 tok/s | 264 × 20μs × 50 = 264ms/s = 26% | 264 × 500ns × 50 = 6.6ms/s = 0.7% |

**设计要点：**
- `generation_` 计数器：每次 `parallel_for` 自增，worker 用 `last_gen != generation_` 判断是否有新任务，避免竞态
- 主线程直接在 `cv_done_` 上 wait，zero-copy 等待
- `n_threads` 默认 `min(8, nproc/2)`：留 4 个超线程给 OS，在 i9-12900K 上实际用 8 线程

```
8 线程下 GEMV 行切分示例（c_attn, out=2304）：
Thread 0: rows [0,   288)  → 读 288×768×2 = 442 KB
Thread 1: rows [288, 576)  → 读 442 KB
...并发发射 8 个内存流 → 饱和 DRAM 多 Bank
```

### 3.3 设计三：`Gpt2Weights` 预建指针表

**消除 热路径中的 264 次 `snprintf` + `unordered_map::find`**

```cpp
struct Gpt2LayerW {
    const float*    ln1g, *ln1b, *ln2g, *ln2b;
    const uint16_t* cattn_w, *cproj_w, *fc_w, *proj_w;
    const float*    cattn_b, *cproj_b, *fc_b, *proj_b;
};
struct Gpt2Weights {
    const uint16_t* wte;  const float* wpe;
    const float*    lnf_g, *lnf_b;
    std::vector<Gpt2LayerW> layers;   // [n_layer]
};
```

`build_weights()` 在 `gpt2_generate()` 入口时**一次性**解析所有权重指针，存入 `Gpt2State`。  
Forward pass 直接访问 `s.w.layers[l].cattn_w`：零字符串操作，L1 cache 友好的 struct-of-pointers。

### 3.4 设计四：GEMV 多级分发器

三层选择逻辑，零运行时分支开销：

```
┌─ 编译期 ─────────────────────────────────────────────┐
│  gemv_f16_avx2.cpp 编译为独立 OBJECT (-mf16c -O3)    │
│  gpt2_model.cpp 不需要任何 SIMD 编译标志              │
└───────────────────────────────────────────────────────┘
         ↓ 链接
┌─ 进程启动期（一次）────────────────────────────────────┐
│  kGemvRange = cpu_features().avx2                     │
│      ? gemv_f16_avx2_range   (F16C + FMA)             │
│      : gemv_f16_scalar_range (通用 fallback)           │
└───────────────────────────────────────────────────────┘
         ↓ 每次调用
┌─ 运行时（每 GEMV）────────────────────────────────────┐
│  if (pool && out >= 128)                              │
│      pool.parallel_for(out, [&](s,e){ kGemvRange }) │
│  else                                                 │
│      kGemvRange(y, W, x, b, 0, out, in)              │
└───────────────────────────────────────────────────────┘
```

## 四、耦合度分析：CPU 优化是否影响 CUDA 扩展？

### 4.1 现有架构中已存在的解耦层

项目设计从一开始就分了清晰的层次：

```
公开 API 层（backend 无关）
  include/axonforge/engine.hpp       Engine / Session
  include/axonforge/models/gpt2.hpp  Gpt2Config / gpt2_generate()
  include/axonforge/backend.hpp      IBackend / IKernel（纯虚接口）

实现层（backend 相关，用户不可见）
  src/models/gpt2_model.cpp          ← 当前耦合点
  src/backend/cpu_x86/               ← x86 专属
  src/core/thread_pool.hpp           ← std::thread，CPU 专属
```

公共头文件（`engine.hpp`, `gpt2.hpp`）**不包含任何 x86 相关内容**，CUDA backend 的用户代码不会被污染。

### 4.2 当前确实存在的耦合问题

`gpt2_model.cpp` 目前同时承担了三件本应分离的事：

```
gpt2_model.cpp
  ├── 计算图定义（做什么：LayerNorm → QKV → Attention → FFN）
  ├── 执行策略（怎么并行：ThreadPool，行切分）
  └── 具体内核（用什么实现：AVX2 F16C，前向声明 x86 函数）
```

这三层混在一个文件里。增加 CUDA backend 时，如果照此写一个 `gpt2_model_cuda.cpp`，会造成**计算图逻辑重复**。

### 4.3 已有的正确抽象：`IBackend` + `IKernel`

`backend.hpp` 已经定义了标准接口：

```cpp
class IBackend {
    virtual bool can_handle(OpType op, ...) = 0;
    virtual std::unique_ptr<IKernel> get_kernel(OpType, ...) = 0;
};
class IKernel {
    virtual void execute(std::span<const Tensor> in,
                         std::span<Tensor>       out,
                         const AttributeMap&    attrs) = 0;
};
```

这正是正确的解耦方向。`Tensor` 已经可以持有 GPU 内存（通过不同的 `TensorStorage` 实现），`IKernel::execute()` 签名对 CPU/CUDA/Vulkan 完全透明。

### 4.4 未来正确的扩展路径

**推荐的三层架构（Phase 2 目标）**：

```
┌─ Layer 1：模型图构建（与 backend 无关）─────────────────────┐
│  gpt2_model.cpp:                                           │
│    for each layer:                                         │
│      backend.dispatch(Op::GEMV_F16, {W, x, b}, {y})       │
│      backend.dispatch(Op::LAYER_NORM, {x, g, b}, {x})     │
│  → 只描述「做什么」，不关心「怎么做」                         │
└─────────────────────────────────────────────────────────────┘
         ↓ virtual dispatch（每 op 1 次）
┌─ Layer 2：IBackend 路由──────────────────────────────────────┐
│  CpuX86Backend::get_kernel(Op::GEMV_F16, F16)              │
│    → GemvF16Avx2Kernel（含 ThreadPool）                     │
│  CudaBackend::get_kernel(Op::GEMV_F16, F16)                │
│    → GemvF16CublasKernel（含 cuBLAS handle）                │
└─────────────────────────────────────────────────────────────┘
         ↓ 实现层
┌─ Layer 3：具体内核──────────────────────────────────────────┐
│  gemv_f16_avx2.cpp  (-mavx2 -mfma -mf16c)                 │
│  gemv_f16_cuda.cu   (-arch=sm_89)                          │
└─────────────────────────────────────────────────────────────┘
```

**这样的好处**：
- 增加 CUDA 只需新建 `src/backend/cuda/`，注册 `CudaBackend`
- `gpt2_model.cpp` **完全不改动**
- ThreadPool 只在 CPU backend 内部使用，不泄漏到模型层

### 4.5 现阶段的工程取舍（以及为什么可以接受）

当前直接耦合是一个**有意识的 Phase 1 权衡**：

| 方面 | 当前（直接耦合） | 理想（IBackend 路由） |
|------|----------------|----------------------|
| 每 GEMV dispatch 开销 | 0 ns（直接函数指针） | ~50–200 ns（虚函数 + map 查找） |
| 增加 CUDA 的代码量 | 需复制计算图逻辑 | 只加内核，不改模型 |
| 当前状态可接受性 | ✅ 性能优先，耦合隔离在 .cpp | — |

对于 GPT-2 117M，每 token 有 264 次 GEMV dispatch，如果每次引入 200 ns 虚函数开销，合计 52 μs/token，在 73 tok/s（每 token 13.5 ms）时占比 0.4%，可以接受。**所以 Phase 2 迁移到 IBackend 路由不会损失性能。**

### 4.6 GTX 1080 Ti 的 CUDA backend 可行性

从刚才 `nvidia-smi` 结果看，机器有 GTX 1080 Ti（sm_61，11 GB GDDR5X，484 GB/s 带宽）。

理论 CUDA 速度估算：
- 权重 247 MB/token，GDDR5X 484 GB/s → 理论 **1959 tok/s**
- 实际受限 kernel launch、CUDA stream 同步：预计 **300–500 tok/s**

但当前 `Tensor` 存储在 mmap CPU 内存，CUDA backend 需要先实现：
1. `CudaTensorStorage`（`cudaMalloc` + `cudaMemcpy`）
2. `cudagemv_f16`（或包装 cuBLAS `cublasHgemv`）
3. `Engine::from_gguf` 支持 `device=cuda`（在加载时 H2D 传输权重）
