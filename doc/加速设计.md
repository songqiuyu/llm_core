# AxonForge 加速架构设计

> 最后更新：2026-05-11  
> 目标：在不紧耦合任何单一模型的前提下，为所有 LLM 推理路径提供可扩展的 SIMD 加速框架。

---

## 一、整体架构

```
┌──────────────────────────────────────────────────────────┐
│                      模型层（Model Layer）                │
│   llama_model.cpp   │   gpt2_model.cpp   │   future...   │
│   调用 gemv_wt()     │   调用 gemv_wt()   │               │
│   可额外实现模型特化算子（如 fused attention）             │
└───────────────┬──────────────────────────────────────────┘
                │  gemv_wt(y, W, x, out, in, pool)
                │  统一接口，按 DType 分发
┌───────────────▼──────────────────────────────────────────┐
│                   分发层（Dispatch Layer）                 │
│  DType::F16   → kLlamaGemvRange  (F16 range, 4-row tile) ✅│
│  DType::Q4_K_M→ kQ4kRange        (Q4K range, 4-row tile) ✅│
│  DType::Q6_K  → kQ6kRange        (Q6K range, 4-row tile) ✅│
│  DType::BF16  → (future)                                  │
└───────────────┬──────────────────────────────────────────┘
                │  运行时选择：启动时检测 cpu_features()
┌───────────────▼──────────────────────────────────────────┐
│                   内核层（Kernel Layer）                   │
│  src/backend/cpu_x86/kernels/                             │
│  ├─ gemv_f16_avx2.cpp    (-mavx2 -mf16c)  ✅ Phase 1/3A │
│  ├─ gemv_q4k_avx2.cpp    (-mavx2 -mf16c)  ✅ Phase 2a/3A│
│  ├─ gemv_q6k_avx2.cpp    (-mavx2 -mf16c)  ✅ Phase 2b/3A│
│  ├─ norm_avx2.cpp         (-mavx2)         ✅             │
│  └─ gemm_int8_vnni.cpp    (-mavx512vnni)   (future)       │
└──────────────────────────────────────────────────────────┘
```

**核心设计原则**：
- **分离编译**：每个 SIMD 内核编译为独立 OBJECT 库（`axonforge_kernels_avx2`），用各自的 `-m` flag，不污染全局编译选项。
- **运行时选择**：`static const FnPtr kXxx = cpu_features().avx2 ? avx2_impl : scalar_impl`，在进程启动时选定，热路径无条件判断。
- **接口不变**：模型层只调用 `gemv_wt()`，无需感知底层是 AVX2 还是标量，新内核对所有模型透明生效。
- **模型特化**：各模型可在分发层之上增加融合算子（如 Flash Attention），不影响通用内核。

## 二、已实现内核

### 2.1 gemv_f16_avx2（F16 权重 × F32 激活）

**文件**：`kernels/gemv_f16_avx2.cpp`，编译选项：`-mavx2 -mfma -mf16c`

**算法**：
```
每 16 个元素（一次主循环迭代）：
  _mm256_cvtph_ps × 2   : 16×F16 → 16×F32  (F16C, 1周期)
  _mm256_fmadd_ps × 2   : 16×FMA             (FMA3, 0.5周期)
  双累加器展开，隐藏 cvtph_ps 的 6 周期延迟
吞吐：~16 元素/3 周期 ≈ 5.3 elem/cycle（vs 标量 ~0.5）
```

**性能**：GPT-2 117M F16 → 73.9 tok/s（+48% vs llama.cpp）

---

### 2.2 gemv_q4k_avx2（Q4_K_M 权重 × F32 激活）✅ Phase 2a

**文件**：`kernels/gemv_q4k_avx2.cpp`，编译选项：`-mavx2 -mfma -mf16c`

**Q4K 块结构**（144 字节 / 256 元素）：
```
[0:1]    d     (f16) 超块 scale
[2:3]    dmin  (f16) 超块 min-scale
[4:15]   scales[12]  8×(6-bit scale, 6-bit min) 打包
[16:143] qs[128]     256 个 4-bit 量化值（每字节 2 个）
```

**算法**（每 256 元素块）：
```
1. F16C 转换 d, dmin → f32
2. 预计算 8 对 (d*scale, dmin*min)，提出乘法出内层循环
3. for chunk in [0..3]:  // 每 chunk = 64 元素
     a. _mm256_loadu_si256  : 一次加载 32 字节（64 个nibble）
     b. AND 0x0F           : 提取低 nibble（元素 0..31）
     c. srli_epi16(4)+AND  : 提取高 nibble（元素 32..63）
     d. 8 组 × 8 元素:
        cvtepu8_epi32 → cvtepi32_ps : nibble 转 f32
        fmsub_ps(d_scale, nibble, dmin_min) : 反量化权重
        fmadd_ps(weight, x, acc)    : 点积累加
   双累加器(acc0/acc1)隐藏 4 周期 FMA 延迟
```

**关键指令**：
```cpp
__m256i lo = _mm256_and_si256(qvec, mask_0F);   // 低 nibble
__m256i hi = _mm256_and_si256(_mm256_srli_epi16(qvec, 4), mask_0F);  // 高 nibble
__m256  w  = _mm256_fmsub_ps(d_scale, nibble_f32, dmin_min);
acc = _mm256_fmadd_ps(w, x_vec, acc);
```

**性能测量（i9-12900K, 8 线程）**：

| 模型 | Phase 0 (标量) | Phase 2a (AVX2) | Phase 2b (+Q6K) | **Phase 3 (+tiling+batch)** |
|------|----------------|-----------------|-----------------|------------------------------|
| TinyLlama-1.1B Q4_K_M | 0.96 tok/s | 4.96 tok/s | ~14.8 tok/s | **~20.1 tok/s** |
| LLaMA-2-7B Q4_K_M | 0.18 tok/s | 0.65 tok/s | ~1.79 tok/s | **~5.0 tok/s** |

## 三、分发机制详解

### 3.1 函数指针选择模式

所有内核通过相同模式在启动时一次性选择，零热路径开销：

```cpp
// F16 (llama_model.cpp / gpt2_model.cpp)
static const GemvRangeFn kLlamaGemvRange =
    cpu_x86::cpu_features().avx2
        ? cpu_x86::gemv_f16_avx2_range
        : gemv_f16_scalar_range;

// Q4K (llama_model.cpp, 任何使用 Q4K 的未来模型可复用)
using Q4kDotRowFn = float(*)(const uint8_t*, const float*, int) noexcept;
static const Q4kDotRowFn kQ4kDotRow =
    cpu_x86::cpu_features().avx2
        ? cpu_x86::q4k_dot_row_avx2
        : q4k_dot_row;
```

### 3.2 ThreadPool 并行策略

```cpp
// gemv_wt 内部：out >= 32 时并行，每个分片调用 kQ4kRange（含 4-row tiling）
if (pool && out >= 32) {
    pool->parallel_for(out, [&](int s, int e) {
        kQ4kRange(y, Wq, x, s, e, in, row_bytes);  // Phase 3A: 4-row tiled
    });
}
```

ThreadPool 为持久 fork-join（唤醒延迟 < 500 ns），避免每次 GEMV 创建线程的开销。

## 四、已完成 Phase 3 加速（2026-05-11）

### Phase 3A：4-row GEMV Tiling（Q4K / Q6K / F16）

**原理**：单行 GEMV 每次需重读输入向量 x（in_dim × 4 字节）。通过同时计算 4 行输出，x 只读一次供 4 个点积使用，x-bandwidth 降低 4×。

```
旧方案（单行）：每行独立读 x[4096] = 16KB
  4096 行 × 16KB = 64MB 的 x 读取（TinyLlama W_Q）

新方案（4-row）：每 4 行共享 x[4096] = 16KB
  1024 组 × 16KB = 16MB 的 x 读取 → 4× 节省
```

额外收益：8 个独立累加器（vs 2 个），更充分利用 FMA 执行单元的 ILP。

**实现**：
- `q4k_dot_4rows_()` — 4 路并行 Q4K dot，共享 x 读取
- `q6k_dot_4rows_()` — 4 路并行 Q6K dot
- `dot_f16_4rows_()` — 4 路并行 F16 dot
- `gemv_*_avx2_range()` 函数先处理 4 倍行组，余数回退单行版本

### Phase 3B：Batched Prefill（llama_forward_batch）

**原理**：原始 `llama_forward_token` 对 T 个 prompt token 串行调用，每次调用都从 DRAM 重新加载同一层的所有权重矩阵（~100MB/layer for LLaMA-2-7B）。

新函数 `llama_forward_batch` 重组循环，**同一权重矩阵的所有 T 次投影连续执行**：

```cpp
// 旧：T × (load W_Q from DRAM, compute)
for t in [0..T): llama_forward_token(token_ids[t], t, state);

// 新：load W_Q into L3 once, compute T times (L3 hits)
for t in [0..T): gemv_wt(Q[t], lw.wq, XB[t], ...);  // W_Q stays in L3
for t in [0..T): gemv_wt(K[t], lw.wk, XB[t], ...);  // W_K stays in L3
// ...
```

对于 TinyLlama（W_Q = 2.25MB < L3 30MB）：token 1..T-1 的 W_Q 来自 L3（~5ns），而非 DRAM（~100ns），TTFT 约改善 6× 数据重用比。

**实测 TTFT 改善**：TinyLlama 2.6×，LLaMA-2-7B 6.6×。

### Phase 3C：AVX2 Attention Dot / SAXPY

Q·K 点积和 V 加权累加替换为 AVX2 FMA 实现：

```cpp
static inline float avx2_dot_f32(const float* a, const float* b, int n);
static inline void  avx2_saxpy  (float* out, float alpha, const float* v, int n);
```

对 HD=64（TinyLlama）：一次 8-wide FMA 迭代完成 8 个元素；全程 8 条指令/head（vs 64 次标量乘加）。

## 五、后续加速计划（Phase 4）

### AVX_VNNI INT8 路径

i9-12900K 支持 AVX_VNNI（256-bit VNNI）。策略：
- 权重量化为 INT8（Q8_0 或动态量化激活）
- `_mm256_dpbusd_epi32`：一条指令完成 32 个 uint8×int8 乘加
- 预期吞吐：~40 tok/s（TinyLlama），~10 tok/s（LLaMA-2-7B）

### Multi-token Decode（Speculative Decoding）

- 草稿模型生成 K 个候选 token，目标模型一次 verify K 个
- 预期 ~3× decode 加速（model-agnostic）

### CUDA Backend（GTX 1080 Ti）

- 目标：~300-500 tok/s（内存带宽 484 GB/s，vs DDR4 51.2 GB/s）
- 实现：`CudaTensorStorage` + `cudagemv_q4k` kernel
- 架构设计：`IBackend::get_kernel()` 路由（已预留 `backend.hpp` 接口）

## 六、模型特化优化（在通用框架上叠加）

通用内核负责所有 GEMV；模型层可额外实现**融合算子**，不破坏通用接口：

| 优化 | 描述 | 适用模型 |
|------|------|----------|
| Fused QKV GEMV | 一次遍历权重完成 Q/K/V 三个投影 | 所有 LLaMA 变体 |
| Flash Attention | 分块 Q·K·V，减少 seq² 内存访问 | 长文本场景 |
| Fused FFN | 将 gate/up 两个 GEMV 合并 | SwiGLU 架构 |
| KV Cache 量化 | 将缓存的 K/V 存为 INT8，节省内存 | 大 context 场景 |

## 八、新增内核 Checklist

添加新的量化格式内核时，遵循以下步骤：

1. **创建内核文件**：`src/backend/cpu_x86/kernels/gemv_<fmt>_avx2.cpp`
   - 实现 `<fmt>_dot_row_avx2(const uint8_t*, const float*, int)` 
   - 实现 `gemv_<fmt>_avx2_range(float*, const uint8_t*, const float*, int, int, int, size_t)`

2. **注册到 CMake**：`src/backend/cpu_x86/CMakeLists.txt` 的 `axonforge_kernels_avx2` 列表

3. **添加分发**：在模型文件中
   - 前向声明（`namespace axonforge::cpu_x86 { ... }`）
   - 定义函数指针（`static const FnType kXxx = cpu_features().avx2 ? avx2_fn : scalar_fn`）
   - 更新 `gemv_wt()` 对应 DType 分支

4. **验证**：`ctest --test-dir build`，确保所有单元测试通过；对比标量输出验证数值正确性
