## 1. 引言

### 1.1 背景

llama.cpp 与其底层的 ggml 张量库已成为端侧 LLM 部署的事实标准。然而，随着端侧 Agent、多模态推理、NPU 异构协同等需求的爆发，这套始于 2023 年初的 C 语言架构正在成为严重瓶颈。

本项目（**AxonForge**）的目标是：**从张量抽象层出发，用现代 C++20 从零构建一个面向端侧全场景的高性能推理引擎**。不追求覆盖所有模型，而是把 Transformer 系 LLM 这一"结构极其规整"的推理场景做到极致。**第一阶段以 x86 Linux 为主战场**，通过插件化 Backend 接口，后续平滑扩展至 CUDA、Vulkan、ARM、NPU 等全平台，无需修改框架核心。

### 1.2 ggml 的核心历史债务（总结）

| 问题类别 | 具体表现 |
|----------|----------|
| **内存管理** | 创建 `ggml_context` 时需手动预计算内存大小；scratch buffer 机制难用易错 |
| **静态计算图** | 序列长度变化时必须重建整棵计算图；KV Cache 碎片整理需重置调度器 |
| **硬件扩展性** | 通过 `#ifdef GGML_USE_CUDA` 堆砌宏定义，新增硬件需侵入修改核心代码 |
| **张量维度限制** | 底层硬编码最多 4 维，高维操作需手动展平/转置 |
| **算子泛化性** | 大量硬编码特定 shape 的 SIMD 汇编，shape 微调即失效 |

## 2. 竞品深度分析

### 2.1 综合对比

| 维度 | ggml | llama.cpp | LiteRT-LM | **llm_core（目标）** |
|------|------|-----------|-----------|---------------------|
| **语言** | C89 + 少量 C++ | C89/C++ 混合 | C++17 + TFLite | C++20 |
| **计算图** | 静态 DAG | 静态 DAG（依赖 ggml） | 静态（TFLite 格式） | **动态 DAG + 符号维度 IR** |
| **内存管理** | 手动预计算 Arena | 继承 ggml | TFLite 自动 | **自动 Liveness Analysis Arena** |
| **Backend 扩展** | `#ifdef` 宏堆砌 | 继承 ggml | Plugin 机制（较完善） | **纯虚接口 + 注册工厂** |
| **量化** | Q4/Q8 系列（优质） | GGUF（标准化好） | INT8/FP16 | **GGUF 兼容 + T-MAC LUT** |
| **KV Cache** | 简单连续 buffer | Paged 支持（有限） | 无 | **Paged + SWA + Hybrid** |
| **多模态** | 扩展中 | llava/clip 支持 | 支持 | 预留扩展接口 |
| **NPU 对接** | 无 | 无（实验性） | 高通 HTP | **DMA-BUF 零拷贝异构** |
| **Android 优化** | 基础 NEON | 基础 NEON | 深度优化 | **T-MAC + DMA-BUF** |
| **目标场景** | 库（嵌入式） | 通用推理服务 | Google 移动端 | **全端侧场景** |

### 2.2 从各项目吸取的优势

**从 ggml 吸取：**
- 紧凑的张量内存布局思想（行主序连续存储）
- 丰富的量化格式设计（Q4_K、Q5_K 等分组量化）
- 轻量级无外部依赖的设计理念

**从 llama.cpp 吸取：**
- GGUF 模型格式（键值元数据 + 张量数据二进制格式，可直接 mmap）
- 完善的 Tokenizer 支持（BPE/Unigram/WordPiece）
- KV Cache 状态管理的实战经验
- 对几十种模型架构的适配经验

**从 LiteRT-LM 吸取：**
- Backend 插件化注册机制的工程实践
- Android HAL 层 NPU 接入方式
- 细粒度 Op 分发的 Dispatcher 设计

**新技术融合：**
- **T-MAC**：基于查表法（LUT）的无乘法 GEMM，彻底消灭反量化开销
- **PagedAttention**：借鉴 vLLM 的 KV Cache 分页思想适配端侧
- **符号维度 IR**：类似 MLIR Affine Dialect，无需重建图即可适应变长序列
- **DMA-BUF / MTLBuffer**：跨 CPU/GPU/NPU 零拷贝异构计算

## 3. 框架定位与核心设计原则

### 框架名称：**AxonForge**（命名空间 `axonforge::`）

> Axon（神经轴突）= 高速信号传导 | Forge（锻造）= 精工出高性能

### 6 大核心原则

1. **现代 C++20 First** — 全面使用 `std::span`、`std::expected`、Concepts、Modules，无裸指针，无 `#ifdef` 硬件路由，零外部依赖（纯 STL + 系统 SIMD 头文件）。

2. **x86 优先，接口全覆盖** — Phase 0/1 深度优化 x86（AVX2/AVX-512/VNNI），但所有硬件相关代码**只能存在于 Backend 实现层**，核心层对 ISA 零感知，ARM/CUDA/NPU backend 不需要改一行 core 代码。

3. **零心智负担的内存管理** — 开发者只需描述计算，内存由引擎基于 Liveness Analysis 自动管理，峰值占用趋近理论最小值。

4. **硬件无关的计算图** — 计算以符号化 DAG IR 表达，Backend 编译为物理执行计划。序列长度变化时只做增量形状推导，不重建图。

5. **插件化 Backend，开闭原则** — 新增硬件支持 = 实现 `IBackend` 接口 + 一行注册代码。框架 core 层代码零修改，新 backend 可作为独立动态库加载。

6. **算子最小完备集 + 运行时融合** — 底层只有 Unary/Binary/Reduce/Movement 四类基础算子，上层算子在 Lowering 阶段展开后由 Fuser 自动融合为硬件优化 Kernel。

## 5. 核心模块设计

### 5.1 Tensor 抽象

#### 设计要点

- **N 维支持**：维度数量无上限（`std::vector<int64_t> shape`），彻底去除 ggml 的 4 维硬限制。
- **类型系统**：枚举涵盖 `F32 / F16 / BF16 / Q8_0 / Q4_K / Q4_0 / I32 / I8 / BOOL`，可通过 Traits 扩展。
- **视图语义（View Semantics）**：`TensorView` 与底层 `TensorStorage` 解耦，reshape/permute/slice 为零拷贝操作，类似 PyTorch 的 `Tensor.view()`。
- **所有权模型**：`TensorStorage` 以 `shared_ptr` 管理底层 buffer，`TensorView` 持有对 storage 的弱引用 + offset + strides，Liveness Analysis 基于 storage 的引用计数触发内存回收。

```
TensorView
  ├── shape:   [batch, seq_len, num_heads, head_dim]
  ├── strides: [计算所得，支持非连续]
  ├── offset:  int64_t（在 storage 中的字节偏移）
  ├── dtype:   DType
  └── storage: shared_ptr<TensorStorage>
                  ├── data: void* （来自 Arena 或 Device 内存）
                  ├── size: size_t
                  └── device: DeviceId
```

#### 关键操作

| 操作 | 实现方式 | 复杂度 |
|------|----------|--------|
| `reshape()` | 修改 shape/strides，不拷贝 | O(1) |
| `permute()` | 修改 strides 排列顺序 | O(1) |
| `slice()` | 修改 offset + shape | O(1) |
| `contiguous()` | 触发实际拷贝，返回新的连续 storage | O(N) |
| `to(device)` | 跨设备迁移，调用 Backend transfer | O(N) |

### 5.2 Memory 子系统

#### 核心组件

```
MemorySystem
  ├── ArenaPool           — 按设备分组的内存池集合
  │     ├── CpuArena      — malloc 对齐分配（64B cache line）
  │     ├── CudaArena     — cudaMallocAsync + Stream-ordered
  │     └── DmaArena      — Android DMA-BUF / Apple MTLHeap
  └── MemoryPlanner       — 基于计算图做 Liveness Analysis
        ├── liveness_map  — tensor_id → [birth_node, death_node]
        └── color_graph() — 贪心着色，复用不重叠生命周期的 slot
```

#### Liveness Analysis 流程

```
1. TopologicalSort(graph)          — 确定节点执行顺序
2. For each tensor t:
     birth[t] = first op that produces t
     death[t] = last op that consumes t
3. GreedyColorAssignment:
     按 birth 顺序遍历，若存在 death[s] <= birth[t] 的已释放 slot s
     则复用 s；否则新增 slot
4. 生成 AllocationPlan: tensor_id → (slot_id, offset, size)
5. Arena 按 plan 一次性分配所有 slot（单次 malloc）
```

**效果**：中间激活的峰值内存接近理论最小值，且对上层开发者完全透明。

### 5.3 计算图 IR

#### IR 节点结构

```
OpNode {
    op_type:   OpType               // MATMUL / ADD / SOFTMAX / ...
    inputs:    vector<TensorId>
    outputs:   vector<TensorId>
    attrs:     AttributeMap         // 算子超参，如 axis, epsilon 等
    meta:      ShapeInferFn         // 形状推导函数
}

ComputeGraph {
    nodes:    vector<OpNode>
    tensors:  unordered_map<TensorId, TensorDesc>  // 含符号维度
    inputs:   vector<TensorId>      // 图的输入
    outputs:  vector<TensorId>      // 图的输出
}
```

#### 符号维度（Symbolic Dimensions）

```cpp
// 维度可以是具体值或符号
using DimExpr = std::variant<int64_t, std::string>;  // e.g., 4096 或 "seq_len"

struct TensorDesc {
    std::vector<DimExpr> shape;
    DType dtype;
};
```

**优势**：推理时序列长度从 `seq_len=1` 变为 `seq_len=512`，引擎仅需将符号代入做 Shape Inference，**无需重建计算图**，相比 llama.cpp 的 `llama_new_context_with_model` 重置方式快 ~100x。

#### Lowering Pipeline（图编译流程）

```
ComputeGraph (High-level, with symbolic dims)
        │
        ▼ [1. ShapeInfer Pass]      符号维度具体化
        ▼ [2. DecomposePass]        高级算子展开为基础算子（如 RoPE → permute + matmul + add）
        ▼ [3. FusePass]             相邻基础算子融合（如 matmul + add + relu → 单 kernel）
        ▼ [4. LayoutPass]           根据 backend 选择最优内存布局（NHWC vs NCHW）
        ▼ [5. AllocationPass]       Liveness Analysis + Arena 分配
        │
        ▼
ExecutionPlan (Backend-specific, concrete shapes, memory addresses)
```

### 5.4 Backend 插件化架构

#### 核心接口（纯虚）

```cpp
// 所有 Backend 均实现此三个接口
class IDeviceAllocator { ... };    // 内存分配/释放/迁移
class IKernelLibrary   { ... };    // 查询/注册算子 kernel
class IBackend         { ... };    // Backend 生命周期 + 执行
```

#### 注册工厂模式

```
BackendRegistry (全局单例)
  ├── register("cpu",    []{ return make_unique<CpuBackend>(); })
  ├── register("cuda",   []{ return make_unique<CudaBackend>(); })
  ├── register("vulkan", []{ return make_unique<VulkanBackend>(); })
  └── register("npu",    []{ return make_unique<QnnBackend>(); })
```

新增 backend 的完整步骤：
1. 创建 `my_backend.h` / `my_backend.cpp`，继承 `IBackend`
2. 在 CMakeLists.txt 添加一个 `target_sources` 行
3. 在初始化代码中 `BackendRegistry::instance().register("my_hw", ...)` — **无需修改任何框架核心代码**

#### 异构调度

```
HeterogeneousPlanner:
  对每个 OpNode，根据以下规则选择 Backend：
  1. 用户偏好注解（@prefer_npu, @prefer_gpu）
  2. Backend capability query（backend.supports(op_type, dtypes)）
  3. 性能模型预测（预先 benchmark 的 op latency table）
  4. 数据迁移代价（若需跨设备，加上 transfer cost）
  → 最终输出 op → backend 的映射表
```

### 5.5 算子系统

#### 最小完备基础算子集

| 类别 | 算子 | 说明 |
|------|------|------|
| **Unary** | `neg, abs, sqrt, exp, log, relu, silu, gelu, sigmoid, cast` | 逐元素 |
| **Binary** | `add, sub, mul, div, pow, max, min` | 逐元素，支持广播 |
| **Reduce** | `sum, mean, max, min` | 沿指定轴归约 |
| **Movement** | `reshape, permute, slice, pad, broadcast, contiguous` | 内存视图操作 |
| **Linalg** | `matmul, batched_matmul` | 核心计算，可接 BLAS/cuBLAS |
| **Attention** | `scaled_dot_product_attention` | 可 fallback 到 Flash Attention |
| **Norm** | `layer_norm, rms_norm` | 可融合进 matmul 后置 |

#### 算子 Fusion 策略（FusePass）

```
规则1: matmul + add              → fused_matmul_bias
规则2: matmul + add + activation → fused_matmul_bias_act
规则3: rms_norm + scale          → fused_rms_norm_scale
规则4: mul + add (residual)      → fused_mul_add
规则5: consecutive unary ops     → element_wise_fusion（单 kernel 多操作）
```

融合规则以 `FusePattern`（模板匹配树）描述，可在不修改 core 代码的情况下通过注册新 Pattern 扩展。

### 5.6 量化子系统

#### 量化格式支持

| 格式 | 比特宽 | 分组大小 | 适用场景 |
|------|--------|----------|----------|
| `Q8_0` | 8-bit | 32 | 精度优先 |
| `Q4_0` | 4-bit | 32 | 平衡 |
| `Q4_K_M` | 4-bit 混合 | 256 | 推荐默认 |
| `Q3_K_M` | 3-bit 混合 | 256 | 内存极限 |
| `Q2_K` | 2-bit | 256 | 超低内存 |
| `LUT4` | 4-bit 查表 | 16 | **T-MAC 专用** |

#### T-MAC LUT GEMM 集成（CPU Backend 核心优化）

传统流程：
```
量化权重 W_q4 ──→ 反量化 W_f32 ──→ GEMM(W_f32, X_f32) → 结果
                   （大量 multiply-add）
```

T-MAC 流程：
```
量化权重 W_q4 ──→ 构建 LUT（每 16 个权重值 → 16 个预算结果）
激活 X_f16    ──→ 查表 + 累加（纯 TBL 指令，无乘法）
                   （ARM NEON TBL / x86 PSHUFB）
```

**集成方案**：在 `CpuBackend` 中为 `matmul(Q4, F16)` 注册专用 kernel，在 Lowering 阶段，当检测到左操作数为 `Q4_K` 或 `LUT4` 格式时，自动路由到 T-MAC kernel，对上层透明。

### 5.7 KV Cache 管理

#### 三种模式

```
KvCacheManager
  ├── ContinuousKvCache    — 连续 buffer，适合单 session 短序列
  ├── PagedKvCache         — 固定大小 page（如 16 tokens），支持多 session 复用
  │     ├── page_size: 16 tokens（可配置）
  │     ├── block_table: session_id → [page_id, ...]
  │     └── free_pages: 空闲 page 池（bitmap 管理）
  └── SlidingWindowKvCache — 仅保留最近 W 个 token 的 KV，适合超长序列
```

#### Paged KV Cache 内存布局

```
每个 Page = [page_size, num_kv_heads, head_dim] 的连续内存块
物理内存：[Page_0][Page_1]...[Page_N]（预分配池）
逻辑地址：通过 BlockTable 动态映射
```

**优势**：
- 无内存碎片，可支持上百个并发 session（端侧 Agent 场景）
- Session 之间可共享相同 prefix 的 Page（Prefix Sharing）
- Page 内存可由 DMA-BUF 分配，实现 CPU/NPU 零拷贝共享

### 5.8 模型加载器

#### 支持格式

| 格式 | 读取方式 | 说明 |
|------|----------|------|
| **GGUF** | `mmap` + 懒加载 | 主要格式，兼容 llama.cpp 生态 |
| **SafeTensors** | `mmap` + JSON header | HuggingFace 标准格式 |
| **ONNX**（预留） | protobuf 解析 | 未来扩展 |

#### GGUF 加载器设计

```
GgufLoader::load(path)
  ├── 1. mmap 文件，解析 header（magic + version）
  ├── 2. 读取 metadata KV（模型架构、超参、tokenizer 配置）
  ├── 3. 读取 tensor 索引（name → offset, shape, dtype）
  ├── 4. 构建 ModelConfig（架构、层数、dim 等）
  ├── 5. 注册所有 TensorView（指向 mmap 地址，零拷贝）
  └── 6. 返回 Model 对象（懒加载：tensor 数据在首次访问时才触发 page fault）
```

### 5.9 采样器流水线

采用**可组合的责任链**设计，每个 Sampler 接收 logits，输出处理后的 logits 或 token id：

```
SamplerPipeline = [
    TemperatureSampler(temp=0.8),
    TopKSampler(k=40),
    TopPSampler(p=0.95),
    RepetitionPenaltySampler(penalty=1.1, context_window=64),
    ArgmaxOrMultinomialSampler()
]
```

每个 Sampler 实现 `ISampler::apply(logits) -> logits` 接口，Pipeline 顺序调用，支持运行时动态组合。

## 6. 关键技术亮点

### 6.1 T-MAC LUT GEMM：突破内存墙

**原理**：将低比特权重（4-bit）的点积运算转化为查表操作。

对于 4-bit 权重，每个权重值只有 16 种可能。对于一个 16 元素的权重分组：
```
W = [w0, w1, ..., w15]，每个 wi ∈ {0,...,15}
X = [x0, x1, ..., x15]，激活值（f16）

预计算 LUT: lut[i] = sum_j(w_j_partial * x_j)  （按位分组）
推理时:     result = TBL(lut, indices)           （ARM NEON vqtbl1q_u8）
```

**实测收益**（来自 T-MAC 论文）：
- Raspberry Pi 5 上 3B 模型：~12 tokens/s（vs ggml 的 ~3 tokens/s）
- 能耗降低约 70%（消除大量乘法运算）

### 6.2 DMA-BUF 零拷贝：突破 I/O 墙

**问题**：CPU → GPU/NPU 数据传输的 memcpy 延迟可占整个推理时延的 20-40%。

**Android 方案**：
```cpp
// 分配物理连续共享内存
int dma_fd = allocate_dma_buf(size);

// CPU 映射
void* cpu_ptr = mmap(nullptr, size, PROT_READ|PROT_WRITE, MAP_SHARED, dma_fd, 0);

// GPU 映射（AHardwareBuffer → OpenCL/Vulkan image）
AHardwareBuffer* ahb = import_from_fd(dma_fd);
VkImage vk_img = import_ahardwarebuffer(ahb);

// NPU 映射（QNN HTP via AHardwareBuffer）
Qnn_Tensor_t npu_tensor = { .memType = QNN_TENSORMEMTYPE_AHWB, .ahwbInfo = {ahb} };
```

**Apple 方案**：`MTLBuffer` 的 Unified Memory 天然支持 CPU/GPU/ANE 零拷贝。

### 6.3 符号维度 IR + 增量形状推导

对比 ggml/llama.cpp 每次序列长度变化都需要重建计算图，llm_core 的符号维度机制：

```
图构建时:  Attention 的 Q shape = [batch, "seq_len", num_heads, head_dim]
推理 1 token: 将 "seq_len" = 1 代入 → ShapeInference 只需 O(num_ops) 时间
推理 512 tokens: 将 "seq_len" = 512 代入 → 同上，无需重建图
```

重建图 vs 增量推导的时间差：对于 32 层 LLaMA-3B 量级，重建图约需 ~10ms，增量推导约 ~0.1ms。

### 6.4 自动算子融合 + Search-based 内核参数优化

```
FusePass 之后的 JIT Kernel 生成：
1. 识别融合模式（matmul + rms_norm + add）
2. 生成融合 kernel 的模板代码（SIMD intrinsic 或 GLSL/MSL shader）
3. 对关键超参（tile_m, tile_n, num_threads）进行小规模网格搜索
4. 缓存最优参数到本地（按设备型号 + op signature）
5. 后续推理直接使用已缓存的最优 kernel
```

### 7.4 使用示例（用户视角）

```cpp
#include "axonforge/engine.hpp"

int main() {
    // 1. 加载模型（GGUF，mmap 懒加载）

        .backend = "cpu",          // 或 "cuda" / "vulkan" / "npu"
        .kv_cache_type = "paged",
        .max_batch_tokens = 2048,
    });

    // 2. 创建会话（拥有独立 KV Cache）
    auto session = engine.new_session();
    auto kv = session.new_kv_state();

    // 3. Tokenize
    auto tokens = engine.tokenizer().encode("你好，世界！");

    // 4. 生成（x86 CPU backend 自动选择 AVX2/AVX-512）
    axonforge::SamplerConfig sampler{.temperature = 0.8f, .top_p = 0.95f};
    for (int i = 0; i < 200; ++i) {
        int32_t next_tok = session.generate_one(tokens, kv, sampler);
        if (next_tok == engine.tokenizer().eos_id()) break;
        std::cout << engine.tokenizer().decode({next_tok});
        tokens = {next_tok};
    }
}
```

## 8. 多平台 Backend 规划

### 8.1 Backend 优先级与技术方案

| Backend | 优先级 | 核心技术 | 适用平台 | 状态 |
|---------|--------|----------|----------|------|
| **x86CpuBackend** | **P0（当前主线）** | AVX2/AVX-512/VNNI、T-MAC LUT GEMM、OpenMP | x86 Linux/Windows | 开发中 |
| **ArmCpuBackend** | P1 | NEON/SVE/DOTPROD、T-MAC LUT GEMM | ARM Linux/Android | 接口预留 |
| **CudaBackend** | P2 | cuBLAS、CUTLASS、Flash Attention 2 | Linux/Windows x86 | 接口预留 |
| **VulkanBackend** | P2 | Vulkan Compute Shader、cooperative_matrix | 跨平台（Linux/Android/Win） | 接口预留 |
| **MetalBackend** | P2 | Metal Performance Shaders、MLX | macOS/iOS Apple Silicon | 接口预留 |
| **QnnBackend** | P3 | QNN HTP、AHardwareBuffer、DMA-BUF | Android Snapdragon | 接口预留 |
| **CoreMLBackend** | P3 | Core ML ANE | iOS/macOS | 接口预留 |
| **OpenClBackend** | P4 | OpenCL 2.0 | 通用嵌入式（RK3588等） | 接口预留 |

### 8.2 x86 CPU Backend 深度设计（P0 主线）

x86 CPU Backend 是 AxonForge 的首个完整实现，也是验证整体架构合理性的基准。

#### ISA 分级与运行时派发

```
x86CpuBackend::init() 时通过 CPUID 检测：
  ┌─ AVX-512F + VNNI + BF16 → 最优路径（Intel Ice Lake+ / AMD Zen4）
  ├─ AVX-512F               → 高性能路径（Intel Skylake-X+）
  ├─ AVX2 + FMA             → 基础路径（绝大多数现代 x86，2013+）
  └─ SSE4.2                 → 兜底路径（不推荐生产使用）

每个算子 Kernel 注册时附带 ISA 需求标签，BackendDispatcher 在运行时
选择满足当前 CPU 的最高级别 Kernel，无需重编译。
```

#### 核心算子优化策略

| 算子 | AVX2 路径 | AVX-512/VNNI 路径 |
|------|-----------|-------------------|
| `GEMM(F32, F32)` | OpenBLAS 或手写 AVX2 tiling | AVX-512 + AMX（预留） |
| `GEMV(Q4_K_M, F32)` | AVX2 LUT GEMV (T-MAC) | AVX-512 VNNI LUT GEMV |
| `GEMM(INT8, INT8)` | AVX2 VNNI（如有） | AVX-512 VNNI |
| `SoftMax` | AVX2 exp 近似 + 水平规约 | AVX-512 掩码向量化 |
| `RMSNorm` | AVX2 向量乘加 | AVX-512 融合 |
| `RoPE` | AVX2 复数乘法 | AVX-512 向量化 |

#### 线程模型

```
OpenMP 或自定义 WorkStealingThreadPool（可选）
  ├── Op 级并行：大 matmul 按 M 维切片，各线程处理一行块
  ├── Layer 级并行：FFN 的两个 Gate 可并行
  └── 线程数 = min(物理核数, max_threads 配置项)
```

### 8.3 插件化接口规范（新 Backend 开发者指南）

新增任意 Backend 只需遵循以下步骤，**核心层代码零修改**：

```
Step 1: 创建 src/backend/my_backend/my_backend.hpp
        继承 axonforge::IBackend，实现所有纯虚函数

Step 2: 在 my_backend.cpp 末尾添加一行：
        AXONFORGE_REGISTER_BACKEND("my_hw", []{ return std::make_unique<MyBackend>(); });

Step 3: CMakeLists.txt 中添加：
        target_sources(axonforge_backends PRIVATE src/backend/my_backend/my_backend.cpp)

Step 4: 用户代码中：
        Engine::from_gguf("model.gguf", {.backend = "my_hw"});
```

### 8.4 异构协同（未来 x86 + CUDA 场景）

```
Prefill 阶段（长 prompt，计算密集）：
    → CUDA Backend 执行所有 Attention + FFN（高并行度）

Decode 阶段（单 token，内存带宽密集）：
    → x86 CPU Backend 执行线性层（T-MAC LUT 极低能耗）
    → 可选：CUDA 执行 Attention（KV Cache 在 GPU 内存）

迁移代价：Tensor.to("cuda") / Tensor.to("cpu")，
          HeterogeneousPlanner 自动在代价最低时触发迁移
```
